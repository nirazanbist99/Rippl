Content Analysis

Technical Description

My content page focuses on "Global Water Accessibility" and examines the challenges faced by developing nations in accessing safe drinking water. The page follows the shared template structure with the standard header, footer, and CSS styles. I used semantic elements such as <article>, <section>, and <aside> to organise the content logically.

Key CSS properties include max-width on the text container for readability, border-left styling on pull-quote boxes to visually distinguish them from body text, and list-style-type on ordered lists for numbered facts. Images use width: 100% and object-fit: cover to remain proportional.

Accessibility

The page follows a strict heading hierarchy from <h1> down to <h3>. All images include descriptive alt attributes. External links are clearly labelled and open in new tabs with target="_blank". Tables (if present) use <th> elements for header rows to help assistive technologies interpret the data.

Challenge

The biggest challenge I had was building the Profile page wizard using only prompt() dialogs. I originally wanted to use HTML modals with custom styling, but the coursework required JavaScript prompt() and confirm() interactions. I learned how to chain multiple prompts inside a loop and handle the case where a user clicks "Cancel" — I used confirm() to ask whether they wanted to skip ahead or stop entirely.

On the Sitemap page, getting the flexible column layout to look balanced was difficult. With four columns containing different numbers of links, some columns were much taller than others. I used min-width: 200px and max-width: 250px to keep all columns roughly the same width, and flex-wrap: wrap ensures they reflow gracefully on narrow screens.

I also found it tricky to keep the progress bar percentage in sync with user actions. Because prompt() is synchronous (it blocks the page until the user responds), I had to call updateProgress() after each individual prompt rather than at the end of each step. This taught me how synchronous browser APIs work differently from asynchronous ones.

Compliance

My pages comply with the Janet Acceptable Use Policy. The Profile page collects answers via JavaScript prompt() dialogs, but this data is only stored temporarily in the browser's memory and displayed on-screen. It is never transmitted to a server, saved in a database, or stored in cookies. When the page is refreshed, all data is lost.

The Sitemap page is purely informational and contains no user-generated content. All links point to pages within the Rippl site. No external tracking scripts, advertisements, or analytics tools are included on any page. Content is educational and relates to SDG 6.

All images are project-generated or sourced from royalty-free platforms and credited appropriately. I validated my HTML and CSS using the W3C validators. The results and screenshots are documented on my validation page.

Accessibility

For the User Profile and Sitemap, I focused on navigating complex structures. The sitemap uses semantic nested lists that are easily traversed by screen readers. For the Profile prompt wizard, I ensured that labels within the prompt text are clear and that users can navigate back to the main site easily if they choose to skip the process. I also used specific focus-styles for sitemap links to improve visual tracking.
